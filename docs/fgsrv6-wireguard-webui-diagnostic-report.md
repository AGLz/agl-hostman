- Configuration: `/etc/wireguard/wg0.conf`

- **Cons**: Requires exclusive port binding, Docker networking complexity

**Conflict**:
- Both are trying to use the same port (51823/udp)
- Only one can run at a time
- Currently: Native is active, container is stopped

## Root Cause Summary

### Problem Chain

1. **Initial Setup**: wg-easy was deployed to provide web UI for WireGuard management
2. **Container Running**: wg-easy was successfully running and managing WireGuard peers
3. **Shutdown Event**: Container received SIGTERM on 2025-10-18 15:11 UTC (reason unknown)
4. **Native WireGuard Active**: Native `wg-quick@wg0` service is already managing the wg0 interface
5. **Port Conflict**: When attempting to restart wg-easy, Docker cannot bind to port 51823/udp
6. **Result**: Container remains stopped, web UI unavailable

### Why Port 51823 is Taken

The native WireGuard interface (`wg0`) is currently running via systemd:
```bash
$ systemctl status wg-quick@wg0
● wg-quick@wg0.service - WireGuard via wg-quick(8) for wg0
   Loaded: loaded
   Active: active (exited)
```

This service binds to UDP port 51823 at the **kernel level**, preventing any other process (including Docker containers) from using it.

## Resolution Options

### Option 1: Disable Native WireGuard, Use Only wg-easy (Recommended for Web UI)

**Steps**:
1. Stop native WireGuard: `systemctl stop wg-quick@wg0`
2. Disable autostart: `systemctl disable wg-quick@wg0`
3. Start wg-easy container: `docker start wg-easy` or `cd /opt/wg-easy && docker compose up -d`
4. Verify web UI: `curl http://localhost:51821`

**Pros**:
- ✅ Web UI becomes available on port 51821
- ✅ Easy peer management through browser
- ✅ All existing peers are already in wg-easy config
- ✅ No configuration migration needed (configs are synchronized)

**Cons**:
- ⚠️ Brief VPN downtime during transition (~30 seconds)
- ⚠️ All management must go through web UI (no manual editing)
- ⚠️ Docker dependency (container must stay running)

---

### Option 2: Keep Native WireGuard, Abandon wg-easy

**Steps**:
1. Remove wg-easy container: `docker compose down` (in `/opt/wg-easy`)
2. Continue using native WireGuard: `systemctl enable --now wg-quick@wg0`
3. Manage peers manually by editing `/etc/wireguard/wg0.conf`
4. Reload config: `wg syncconf wg0 <(wg-quick strip wg0)`

**Pros**:
- ✅ No disruption (current state)
- ✅ Better performance (kernel-level, no Docker overhead)
- ✅ More control over configuration
- ✅ No dependency on Docker or web services

**Cons**:
- ❌ No web UI (manual configuration only)
- ❌ Steeper learning curve for peer management
- ❌ More error-prone (syntax mistakes in config files)

---

### Option 3: Change wg-easy to Use Different Port (Complex)

**Steps**:
1. Modify docker-compose.yml to use different WireGuard port (e.g., 51824)
2. Update `WG_PORT=51824` environment variable
3. Update firewall rules to allow 51824/udp
4. Reconfigure all clients to connect to new port
5. Start wg-easy container

**Pros**:
- ✅ Both native and container can coexist
- ✅ Web UI available for managing container-based VPN

**Cons**:
- ❌ Two separate WireGuard networks (complexity)
- ❌ Duplicate configuration management
- ❌ All clients must be reconfigured (disruptive)
- ❌ Not recommended - adds unnecessary complexity

---

### Option 4: Use wg-easy as Management UI Only (Advanced)

**Concept**: Run wg-easy temporarily when management is needed, then stop it.

**Steps**:
1. When management needed: `systemctl stop wg-quick@wg0 && docker start wg-easy`
2. Perform web UI changes (add/remove peers)
3. Export config from wg-easy to native config
4. Stop container and restart native: `docker stop wg-easy && systemctl start wg-quick@wg0`

**Pros**:
- ✅ Best of both worlds (performance + UI)
- ✅ Minimal downtime (only during transitions)

**Cons**:
- ❌ Manual process (not automated)
- ❌ Requires config synchronization
- ❌ Higher complexity
