## Problem Statement

O administrador do parque tecnológico AGL (eu) perde ~30 min/dia em verificações manuais de estado dos CTs/VMs Proxmox. Apesar de ter PegaProx (CT210) e OpenClaw (CT179), ainda não existe uma visão consolidada e visual do estado da infraestrutura.

**Dor específica:** CTs "saudáveis" que na verdade têm problemas de configuração — só detetáveis com "olho humano" nos logs ou comandos manuais.

**Objetivo:** Dashboard Laravel com grid visual de 20+ CTs/VMs, cores semáforo (verde/amarelo/vermelho), mostrando CPU%, RAM%, disco e estado.

## What Makes This Cool

Um único ecrã onde vês todo o parque AGL de relance. Verde = siga, Amarelo = atenção, Vermelho = ação. O "aha!" é quando detetas um problema que o Proxmox não mostra explicitamente.

## Constraints

- Usar Laravel existente em `src/`
- Docker Compose para orquestração (path para cluster depois)
- Polling 30-60s aceitável para v1
- MVP em 2-3 dias, não 2 semanas

## Approaches Considered

### Approach A: PegaProx Bridge (ESCOLHIDO)
Endpoint PHP Laravel → proxy PegaProx → frontend Inertia React. SQLite cache 60s.

**Pros:** Rápido (2-3 dias), usa PegaProx existente, Laravel+Inertia já no projeto  
**Cons:** Depende do PegaProx, não escala para 100+ CTs sem paginação, healthcheck limitado

### Approach B: Metrics Gateway (arquitetura ideal)
Serviço Node.js separado que faz poll Proxmox + healthchecks, API REST para Laravel.

**Pros:** Separação concerns, healthchecks nativos, preparado para cluster  
**Cons:** Mais infra, 1-2 semanas de trabalho

### Approach C: Agentless Discovery (lateral)
Aproveitar OpenClaw existente como "agente", ler JSON de healthchecks.

**Pros:** Zero novos containers, usa OpenClaw que já monitoriza  
**Cons:** Acoplado ao OpenClaw, delay até 5min, arquitetura menos pura

## Recommended Approach

**Approach A: PegaProx Bridge**

**Razão:** Entrega valor em 2-3 dias. O PegaProx já resolve o problema difícil (auth Proxmox). Healthchecks podem ser adicionados na v2 via Approach C como extensão. Foco no dashboard, não em infra nova.

**Completeness:** 7/10 — happy path coberto, edge cases na v2

### Componentes

1. **Backend (Laravel PHP)**
   - `App\Http\Controllers\Dashboard\InfraController`
   - Service: `App\Services\PegaProxService` (com cache 60s)
   - Job: `FetchProxmoxMetrics` (queue:default)
   - Model: `InfraMetric` (SQLite, TTL 5min)

2. **Frontend (Inertia React)**
   - Componente: `resources/js/Pages/Dashboard/InfraGrid.tsx`
   - Grid responsivo: cards ou tabela com cores semáforo
   - Polling: 45s via `setInterval` + Inertia reload

3. **Configuração**
   - `config/services.php`: URL do PegaProx, credenciais
   - `.env`: `PEGAPROX_URL`, `PEGAPROX_TOKEN`

### Métricas e Cores

| Métrica | Verde | Amarelo | Vermelho |
|---------|-------|---------|----------|
| Estado CT | running | stopped | locked/error |
| CPU% | < 70% | 70-90% | > 90% |
| RAM% | < 80% | 80-95% | > 95% |
| Disco% | < 80% | 80-90% | > 90% |

**Algoritmo de cor:** A pior métrica determina a cor do card.

### Endpoints

```
GET /dashboard/infra       → Página Inertia com grid
GET /api/infra/metrics     → JSON métricas (para AJAX polling)
POST /api/infra/refresh    → Forçar refresh do cache
```

### Estrutura do PegaProxService

```php
class PegaProxService
{
    // Auth
    public function login(string $user, string $pass): string; // session cookie
    
    // Endpoints
    public function getClusters(): array;
    public function getVms(string $clusterId): array; // VMs + CTs
    public function getNodeMetrics(string $clusterId, string $node): array;
}
```

### Fluxo de Dados

```
1. Laravel → POST /api/auth/login → session cookie
2. Laravel → GET /api/clusters → [cluster_id]
3. Laravel → GET /api/clusters/{id}/vms → [vms_data]
4. Cache em SQLite (TTL 60s)
5. Frontend React → polling 45s → Laravel API
```
