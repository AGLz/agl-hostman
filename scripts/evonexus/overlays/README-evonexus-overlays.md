# Overlays EvoNexus (CT242 / `/opt/evonexus`)

Ficheiros montados por volume em `docker-compose.hub.yml` no serviço **dashboard**:

| Ficheiro em `/opt/evonexus` | Destino no contentor |
|-----------------------------|----------------------|
| **`server.js.atlas-result-text`** (conteúdo = `terminal-server-server.js` do repo) | `terminal-server/src/server.js` |
| `terminal-server-provider-config.js` | `terminal-server/src/provider-config.js` |
| `claude-bridge.js` | `terminal-server/src/claude-bridge.js` |
| `settings.json.uv-hooks` | `/workspace/.claude/settings.json` |

**Deploy do `server.js`:** no CT242 actual, o `docker-compose.hub.yml` usa o ficheiro host **`./server.js.atlas-result-text`** montado em `terminal-server/src/server.js`. Copiar o overlay canónico do repo para esse nome no host, por exemplo: `pct push 242 …/terminal-server-server.js /opt/evonexus/server.js.atlas-result-text` (a partir do **fgsrv7**). O ficheiro **`terminal-server-server.js`** no repo é a fonte legível / versionada do mesmo patch.

**Variáveis no compose (dashboard):** `IS_SANDBOX=1` — o `claude-bridge` repassa ao processo `claude` e permite `--dangerously-skip-permissions` mesmo com UID 0 quando combinado com `global_settings.dangerouslySkipPermissions` em `providers.json`.

**Docker + Tailscale (CT242):** o **host** do CT tem Tailscale (`tailscale0`, ex. `100.93.121.73`). Os contentores estão na bridge **`evonexus_default`** (`172.18.0.0/16`) e **já alcançam** IPs da tailnet via NAT do host — teste: `docker exec evonexus-dashboard curl -sS -o /dev/null -w "%{http_code}" http://100.125.249.8:4000/health` → **401** (ligação OK). **Não** instalar um segundo `tailscaled` dentro do contentor: modo **userspace-networking** não cria `tailscale0`; só expõe SOCKS5 (`127.0.0.1:1055`) e regista outro nó (ex. `fgsrv07-evonexus-dash`). Apps sem `HTTP_PROXY` ignoram esse SOCKS e parecem “sem Tailscale”. Preferir **`.env`** com `LITELLM_GATEWAY_URL` / `ANTHROPIC_BASE_URL` = IP:porta correctos na tailnet (ex. LiteLLM **`100.125.249.8:4000`**, não `100.94.221.87:4000` se nada escuta na 4000). LAN **`192.168.0.0/24`** a partir do CT242 costuma **timeout** sem subnet router Tailscale — usar IP Tailscale do serviço ou `tailscale up --advertise-routes=…` noutro nó. Se precisares de **toda** a tailnet no dashboard sem proxy: `network_mode: host` no serviço `dashboard` (trade-off: portas no host).

**Terminal Jarvis e stack de pedidos:** o processo `claude` no EvoNexus usa sempre a **API Anthropic** (`ANTHROPIC_BASE_URL` → LiteLLM `/v1/messages`). Aliases cujo upstream é **OpenAI** (`gpt-*`, etc.) são **traduzidos** nesse caminho; é comum ver **turnos sem texto** (“Crunched…”, “Churned…”, stream vazio) após **Read** ou outras ferramentas — não é necessariamente URL errada; ver issues LiteLLM / compatibilidade Anthropic↔OpenAI em stream. O **`sync-providers-anthropic-from-env.py`** normaliza `ANTHROPIC_BASE_URL` / `LITELLM_GATEWAY_URL` removendo sufixo **`/v1`** se existir (evita path duplicado).

**Modelo / orquestração:** o **default** EvoNexus (quando `.env` não define `ANTHROPIC_MODEL` / `EVONEXUS_ANTHROPIC_MODEL`) é aplicado por **`sync-providers-anthropic-from-env.py`** — **`qwen3.5-plus`** (DashScope via LiteLLM; mais estável no terminal após ferramentas; confirmado em smoke CT242 2026-05). O agente **`jarvis.md`** alinha o frontmatter `model:` ao mesmo. O script **`run-ct242-claude-smoke-matrix.sh`** usa por defeito só **`qwen3.5-plus`** (exportar **`EVONEXUS_SMOKE_MODELS`** para regressão multi-modelo). Para **GPT** no terminal pode definir-se no `.env` `ANTHROPIC_MODEL=gpt-5.4-mini` (aceitar risco de turnos vazios) ou preferir **`glm-5`** / **`claude-*`** conforme gateway. Com **`qwen-coder`** o CLI por vezes **repete frases** ou deixa o turno incompleto; **`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`** está **`0`** no overlay.

**E2E + monitorização de chamadas (repo):** na raiz do agl-hostman, **`npm run test:e2e:evonexus`** corre Playwright (`playwright.evonexus.config.js`) sobre **`tests/e2e/evonexus-terminal-ws.spec.js`**. O teste anexa **`evonexus-network-monitor`** (JSON): pedidos/respostas HTTP para `*aglz.io*` (até 100 entradas) e eventos WebSocket do terminal (abertura, até 5 frames in/out com pré-visualização, fecho com contagens). A lógica de serialização está em **`tests/e2e/helpers/evonexus-network-monitor.js`** e tem teste unitário **`tests/unit/evonexus-network-monitor-payload.test.js`**. Se `npx playwright install` falhar ou o Chromium morrer com **`Permission denied`** em `.so` do browser, o `$HOME/.cache` ou o mount do projecto pode ter **`noexec`**: usar por exemplo **`PLAYWRIGHT_BROWSERS_PATH=/var/tmp/pw-browsers`** (directório local com exec) antes de `playwright install` e do mesmo env ao correr o e2e.

**Probe de modelos + `--debug-file` (agl-hostman):** o script **`scripts/evonexus/claude-gateway-model-probe.py`** corre **dentro** do contentor `evonexus-dashboard` (`cwd` `/workspace`), lê credenciais só de `config/providers.json`, e para cada modelo corre `claude -p … --model … --dangerously-skip-permissions --debug-file /tmp/claude-probe-logs/…`. Lista por defeito inclui **`glm-5-turbo`**, **`glm-5`**, **`glm-4.7`**, **`glm-4.7-flash`**, **`gpt-5.5`**, **`gpt-5.4`**, **`gpt-5.4-mini`**, **`claude-opus-4-7`**, **`deepseek-4`**, **`qwen3.5-plus`**, **`qwen-coder`** (aliases definidos em **`config/litellm/config.yaml`**). **`QUICK_PROBE=1`** (`docker exec -e QUICK_PROBE=1 …`) limita ao prompt **`no_tools`** (rápido para validar `/v1/models` após deploy). Para vários **`--agent`**, **`scripts/evonexus/claude-agents-smoke.py`** (vários modelos e prompt via env: **`EVONEXUS_SMOKE_MODELS`**, **`EVONEXUS_SMOKE_AGENTS`**, **`EVONEXUS_SMOKE_PROMPT`**). Orquestração no host com **`pct`**: **`scripts/evonexus/run-ct242-claude-smoke-matrix.sh`** (por defeito só **`qwen3.5-plus`** — alinhado ao default EvoNexus; exportar **`EVONEXUS_SMOKE_MODELS`** para matriz multi-modelo). Deploy típico manual: `scp` → `fgsrv7` → `pct push 242 … /opt/evonexus/` → `docker cp … evonexus-dashboard:/tmp/` → `docker exec -w /workspace evonexus-dashboard python3 /tmp/claude-gateway-model-probe.py`. **Após adicionar aliases no repo:** aplicar `config/litellm/config.yaml` no host do gateway LiteLLM e **reiniciar / reload** o proxy — senão o Claude Code devolve **`Invalid model name`** para nomes ainda não registados.

**Catálogo LiteLLM / custo (referência):** índice de providers — [Providers | LiteLLM](https://docs.litellm.ai/docs/providers/); catálogo — [models.litellm.ai](https://models.litellm.ai/). Com **Z.AI**, **Anthropic**, **OpenAI** e **Alibaba (DashScope)** activos: **baratos / volume** — `glm-4.7-flash`, `glm-flash`, `qwen3.5-flash`, `qwen-turbo`, `gpt-5.4-nano`, `gpt-5-nano`, `deepseek-4` (alias → V4 Flash), `deepseek`, entradas **OpenRouter `:free`** no `config.yaml`; **qualidade / agentes** — `glm-5`, `glm-5-turbo`, `qwen3.5-plus`, `gpt-5.4-mini`, `gpt-5.4`, `claude-opus-4-7`, `claude-sonnet-4-6`. **Nota:** **`deepseek-4`** no repo é **alias** para **`deepseek/deepseek-v4-flash`** (não confundir com naming interno DeepSeek).

**Matriz rápida (2026-05-14, `QUICK_PROBE=1` no CT242, gateway *antes* de reload com aliases novos):** `glm-5`, `glm-4.7-flash`, `gpt-5.5`, `claude-opus-4-7`, `qwen3.5-plus`, `qwen-coder` → **OK**; `glm-4.7` → resposta vazia / proxy (HTTP 200 malformado); `glm-5-turbo`, `gpt-5.4`, `gpt-5.4-mini`, `deepseek-4` → **`Invalid model name`** até o LiteLLM servir o `config.yaml` actualizado.

**Read + Claude aliases (teste anterior):** **`qwen3.5-plus`** respondeu bem após **Read** em **`CLAUDE.md`**; **`qwen-coder`** / **`qwen-turbo`** muitas vezes **stdout mínimo**; **`claude-sonnet-4-6`** / **`claude-haiku-4-5-20251001`** → **`Execution error`** nesse cenário (streaming após ferramenta); o mesmo par **passou** com **`--agent jarvis`** sem obrigar Read.

**Modelo barato (Anthropic + LiteLLM):** correr dentro do contentor `python3 /opt/evonexus/sync-providers-anthropic-from-env.py` (cópia em `/tmp` + `docker cp`) ou definir no `.env` do volume `ANTHROPIC_MODEL` / `EVONEXUS_ANTHROPIC_MODEL`. O script preenche omissões com **`qwen3.5-plus`**. Para gateway LiteLLM, mantém **`ANTHROPIC_AUTH_TOKEN`** e evita o aviso *Auth conflict* do Claude Code 2.1+ ao remover `ANTHROPIC_API_KEY` quando ambos estavam definidos para o mesmo proxy. Se `ANTHROPIC_BASE_URL` não for `api.anthropic.com`, o sync define **`API_TIMEOUT_MS=900000`** (15 min) salvo override no `.env`; a whitelist do overlay inclui esta chave para o spawn receber o valor.

**Persona do agente no gateway:** o `claude-bridge` injeta **`--system-prompt`** com o corpo do `.claude/agents/<agent>.md` (sem frontmatter YAML; CRLF-safe) quando o provider é Anthropic e o `ANTHROPIC_BASE_URL` é um gateway. **Não** se passa **`--agent`** nesse modo — combinar os dois fazia o CLI duplicar persona e o modelo (ex. Qwen via LiteLLM) muitas vezes **parava ou devolvia saída vazia** após a primeira ferramenta.

**Sessões PTY (`terminal-server-server.js`):** ao criar sessão nova (`POST /api/sessions/create` ou ramo de criação em `for-agent`), as sessões **inativas** do mesmo agente + `workingDir` ficam com **`superseded: true`**. Se o WebSocket ainda tiver um `sessionId` antigo e enviar `start_claude`, o servidor **redireciona** para a sessão mais recente não obsoleta — evita corrida em que o PTY antigo recebia `SIGTERM` / `read EIO` e o Claude “parava” após a primeira ferramenta.

**Chat vs terminal (`cwd`):** o modo **chat** (`chat-bridge.js`) usa `cwd = session.workingDir || process.cwd()`. Se o SPA ou dados legacy gravarem **`/workspace/workspace`** (espelho sob `/workspace`, referenciado em scripts como `unify-single-sqlite-evonexus-db.sh`), o Agent SDK corria nesse subtree enquanto o projeto útil está em **`/workspace`** — **Glob/Read** falhavam; o PTY podia parecer “melhor” por contexto interactivo. O overlay **colapsa** `/workspace/workspace/…` → **`/workspace/…`** ao criar sessão, ao **carregar** sessões persistidas e ao **`join_session`** no WebSocket. Opcional: **`WORKSPACE_ROOT`** ou **`EVONEXUS_WORKSPACE_ROOT`** fixam a fronteira em vez da derivação `../../../` a partir de `terminal-server/src/server.js`.

**Paragem após Read (diagnóstico em produção):** nos logs do `evonexus-dashboard`, se aparecer **`Claude session <id> error: read EIO`** e **`exited with code 143`** pouco depois de **`Claude session <outro-id> started successfully`**, o PTY antigo foi **terminado** (`SIGTERM`). Causa típica: **`DELETE /terminal/api/sessions/<id-antigo>`** (o log de acesso Flask mostra-o mesmo quando não carregaste no botão apagar — o fluxo **“nova sessão”** no SPA pode disparar esse DELETE em segundo plano). O overlay recusa **`DELETE`** com **`409 session_active`** enquanto `claude` estiver activo nessa sessão, salvo **`?force=1`** (remoção explícita com processo a correr).

**Testes manuais:** abrir **nova sessão** e em seguida **apagar a sessão anterior** com o Claude **ainda a correr** nessa sessão reproduz de propósito o cenário acima (SIGTERM no PTY antigo). Para testar o Jarvis de forma válida: usar **uma só sessão**, ou **parar** o terminal na sessão antiga antes de `DELETE`, ou apagar a antiga **só quando já não estiveres a pedir prompts** nela. Para apagar à força: **`DELETE .../sessions/<id>?force=1`**.

**Hooks (`settings.json.uv-hooks`):** canónico no repo em **`scripts/evonexus/overlays/settings.json.uv-hooks`**. Comandos usam **`cd /workspace`** e **`bash /workspace/.claude/hooks/...`**. No Claude Code, o campo **`timeout`** dos hooks está em **segundos** (doc oficial); valores como `120000` seriam centenas de horas — usar **`120`**–**300** conforme necessidade. Os hooks **`uv run … claude_hook_dispatcher.py`** usam **`"async": true`** para não bloquear o turno do modelo enquanto o `uv` resolve o ambiente (evita “Brewed” longo / prompt vazio após a 1.ª ferramenta). O `agent-tracker` no matcher **`Agent`** também é **`async: true`** — um tracker síncrono a falhar ou a demorar bloqueava a delegação (`Task`/`Agent`) e o Jarvis repetia desculpas sem chamar sub-agentes. **`permissions.allow`** inclui **`Bash(*)`** (workspace Evo como fronteira de confiança), **`Glob`**, **`Grep`**, **`Task`**, **`Agent`**, **`Skill`**, **`WebFetch`**, **`WebSearch`**, com **`deny`** mínimo em comandos destrutivos. O `agent-tracker` em **Stop** mantém-se síncrono com **`timeout`** curto (**15** s). O `claude-bridge` define **`CLAUDE_PROJECT_DIR`** no spawn (última chave do `env`) para o `agent-tracker` e paths canónicos sob `/workspace`.

Fonte canónica no repo: `scripts/evonexus/overlays/` e `scripts/evonexus/sync-providers-anthropic-from-env.py`.

**Deploy CT242:** `scripts/evonexus/deploy-adw-routines-ct242.sh` (correr no **host Proxmox** com `pct`; staging em `/opt/evonexus` + `tar` para o CT). Rotinas correm com `uv run python3` dentro do contentor.

## Agente Jarvis no menu `/agents`

O endpoint `/api/agents` lista apenas ficheiros **`/workspace/.claude/agents/*.md`**. Memória em `agent-memory/jarvis/` **não** cria entrada no menu — é necessário **`jarvis.md`** nessa pasta.

- **Definição canónica:** `scripts/evonexus/agents/jarvis.md` (repo agl-hostman).
- **Persistência após `docker compose pull`:** montar read-only como o Atlas, por exemplo no `docker-compose.hub.yml` do dashboard:  
  `- ./jarvis.md:/workspace/.claude/agents/jarvis.md:ro`  
  com cópia do ficheiro em `/opt/evonexus/jarvis.md`.
- **RBAC por camada (`agent_access.mode: layer`):** em `dashboard/backend/models.py`, o mapa **`AGENT_LAYERS`** tem de incluir `"jarvis": "engineering"` (ou outra camada); sem isso o agente aparece **bloqueado** para papéis que não sejam `admin`/`all`. Após editar `models.py` no contentor, `docker compose restart dashboard`. O ficheiro vem da **imagem** — após `docker compose pull` / recreate, voltar a correr **`scripts/evonexus/patch-dashboard-models-jarvis-layer.py`** dentro do contentor (ou cópia em `/opt/evonexus/` no CT242).
