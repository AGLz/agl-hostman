# Database High Availability

## Overview

Database high availability (HA) ensures your database layer remains operational with minimal downtime during failures, maintenance, or disasters. This skill covers replication, failover, load balancing, and disaster recovery for MySQL, PostgreSQL, and Redis.

### Key Metrics

| Metric | Target | Description |
|--------|--------|-------------|
| **RTO** | < 5 minutes | Recovery Time Objective - time to restore service |
| **RPO** | < 1 minute | Recovery Point Objective - max data loss |
| **Uptime** | 99.95% | ~22 minutes downtime/month |
| **Failover** | < 30 seconds | Automatic promotion time |

### Architecture Patterns

```
Application Layer
      ↓
Load Balancer (ProxySQL/PgBouncer/Twemproxy)
      ↓
┌─────────┬─────────┬─────────┐
│ Primary │ Replica │ Replica │  ← Synchronous Replication
└─────────┴─────────┴─────────┘
      ↓           ↓           ↓
   Storage     Storage     Storage  ← Separate storage nodes
```

## MySQL Replication

### Master-Slave Replication

**Setup Steps:**

1. Configure master binary logging
2. Create replication user
3. Take consistent backup
4. Configure slave with master position
5. Start replication threads

**Configuration:**

```sql
-- Master: /etc/mysql/mysql.conf.d/mysqld.cnf
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog-format = ROW
binlog-do-db = laravel
gtid-mode = ON
enforce-gtid-consistency = ON
sync-binlog = 1

-- Slave: /etc/mysql/mysql.conf.d/mysqld.cnf
[mysqld]
server-id = 2
relay-log = mysql-relay-bin
read-only = 1
super-read-only = 1
gtid-mode = ON
enforce-gtid-consistency = ON
```

**Replication User:**

```sql
CREATE USER 'replicator'@'%' IDENTIFIED WITH mysql_native_password BY 'strong_password';
GRANT REPLICATION SLAVE ON *.* TO 'replicator'@'%';
FLUSH PRIVILEGES;
```

**Change Master:**

```sql
CHANGE MASTER TO
  MASTER_HOST='master.example.com',
  MASTER_USER='replicator',
  MASTER_PASSWORD='strong_password',
  MASTER_AUTO_POSITION=1;

START SLAVE;
```

### GTID (Global Transaction Identifiers)

GTID simplifies replication management and failover:

```sql
-- Check GTID status
SHOW MASTER STATUS;
SHOW SLAVE STATUS\G

-- Skip transaction on error (use carefully)
SET GTID_NEXT='<gtid>';
BEGIN; COMMIT;
SET GTID_NEXT='AUTOMATIC';
START SLAVE;
```

### Monitoring Replication

```sql
-- Replication health check
SELECT
  SLAVE_IO_STATE,
  SLAVE_IO_RUNNING,
  SLAVE_SQL_RUNNING,
  SECONDS_BEHIND_MASTER
FROM SHOW SLAVE STATUS;
```

**Thresholds:**
- SLAVE_IO_RUNNING: Yes
- SLAVE_SQL_RUNNING: Yes
- SECONDS_BEHIND_MASTER: < 5 seconds

## PostgreSQL Replication

### Streaming Replication

**Master Setup:**

```sql
-- /etc/postgresql/15/main/postgresql.conf
wal_level = replica
max_wal_senders = 5
max_replication_slots = 5
hot_standby = on
synchronous_commit = on
synchronous_standby_names = 'standby1,standby2'

-- /etc/postgresql/15/main/pg_hba.conf
host    replication     replicator      10.0.0.0/8      md5
```

**Replication User:**

```sql
CREATE USER replicator WITH REPLICATION ENCRYPTED PASSWORD 'strong_password';
```

**Slave Setup:**

```bash
# Stop PostgreSQL
sudo systemctl stop postgresql

# Clean data directory
sudo rm -rf /var/lib/postgresql/15/main/*

# Base backup from master
pg_basebackup -h master.example.com -D /var/lib/postgresql/15/main \
  -U replicator -P -v -R -X stream -C -S slot1

# Start PostgreSQL
sudo systemctl start postgresql
```

**Standby Signal:**

```sql
-- /etc/postgresql/15/main/recovery.conf (auto-generated by pg_basebackup)
standby_mode = 'on'
primary_conninfo = 'host=master.example.com port=5432 user=replicator'
```

### Logical Replication

For selective table replication:

```sql
-- Publisher
CREATE PUBLICATION laravel_pub FOR TABLE users, tasks, projects;

-- Subscriber
CREATE SUBSCRIPTION laravel_sub
CONNECTION 'host=master.example.com dbname=laravel user=replicator'
PUBLICATION laravel_pub
WITH (copy_data = true);
```

### Monitoring

```sql
-- Replication lag
SELECT
  now() - pg_last_xact_replay_timestamp() AS replication_lag;

-- WAL size
SELECT
  pg_size_pretty(sum(pg_walfile_name_offset(lsfn).offset)) AS wal_size
FROM pg_ls_waldir() AS lsfn;
```

## Redis Cluster

### Sentinel (High Availability)

**Sentinel Configuration:**

```ini
# /etc/redis/sentinel.conf
port 26379
sentinel monitor mymaster 10.0.0.10 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel parallel-syncs mymaster 1
sentinel failover-timeout mymaster 10000
sentinel auth-pass mymaster strong_password
```

**Setup:**

```bash
# Install Redis Sentinel
sudo apt install redis-sentinel

# Configure 3+ sentinel instances
sudo cp /etc/redis/sentinel.conf /etc/redis/sentinel1.conf
sudo cp /etc/redis/sentinel.conf /etc/redis/sentinel2.conf
sudo cp /etc/redis/sentinel.conf /etc/redis/sentinel3.conf

# Start all sentinels
sudo systemctl start redis-sentinel@{1,2,3}
```

**Testing Failover:**

```bash
# Simulate master failure
redis-cli -h 10.0.0.10 -p 6379 DEBUG SLEEP 30

# Check sentinel logs
sudo journalctl -u redis-sentinel -f
```

### Cluster Mode (Sharding)

**Cluster Setup:**

```bash
# Create 6 instances (3 masters, 3 replicas)
for port in 7000 7001 7002 7003 7004 7005; do
  mkdir -p /etc/redis/cluster/$port
  cat > /etc/redis/cluster/$port/redis.conf <<EOF
port $port
cluster-enabled yes
cluster-config-file nodes-$port.conf
cluster-node-timeout 5000
appendonly yes
EOF
  redis-server /etc/redis/cluster/$port/redis.conf &
done

# Create cluster
redis-cli --cluster create \
  10.0.0.10:7000 10.0.0.11:7001 10.0.0.12:7002 \
  10.0.0.13:7003 10.0.0.14:7004 10.0.0.15:7005 \
  --cluster-replicas 1
```

**Cluster Operations:**

```bash
# Check cluster status
redis-cli -c -p 7000 cluster info

# List nodes
redis-cli -c -p 7000 cluster nodes

# Add new node
redis-cli --cluster add-node 10.0.0.16:7006 10.0.0.10:7000

# Reshard slots
redis-cli --cluster reshard 10.0.0.10:7000
```

## Load Balancing

### ProxySQL for MySQL

**Installation:**

```bash
# Install ProxySQL
sudo apt install proxysql

# Configure admin interface
sudo mysql -h 127.0.0.1 -P 6032 -u admin -padmin
```

**Configuration:**

```sql
-- Add MySQL servers to hostgroup
INSERT INTO mysql_servers(hostgroup_id, hostname, port) VALUES
  (10, 'mysql-master.example.com', 3306),
  (20, 'mysql-slave1.example.com', 3306),
  (20, 'mysql-slave2.example.com', 3306);

-- Query rules for read/write splitting
INSERT INTO mysql_query_rules(active, match_pattern, destination_hostgroup, apply) VALUES
  (1, '^SELECT.*FOR UPDATE', 10, 1),
  (1, '^SELECT', 20, 1),
  (1, '.*', 10, 1);

-- Load to runtime
LOAD MYSQL SERVERS TO RUNTIME;
LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL SERVERS TO DISK;
SAVE MYSQL QUERY RULES TO DISK;
```

**Monitoring:**

```sql
-- Check server status
SELECT * FROM stats_mysql_connection_pool;

-- Query metrics
SELECT * FROM stats_mysql_query_digest ORDER BY sum_time DESC LIMIT 10;
```

## Failover Automation

### MySQL Failover with MHA

**Setup MHA (Master High Availability):**

```bash
# Install MHA Manager
sudo apt install mha4mysql-manager

# Configure masterha_default.cnf
[server default]
user=mha_manager
password=mha_password
ssh_user=root
repl_user=replicator
repl_password=replication_password
ping_interval=1

[master1]
hostname=master.example.com
candidate_master=1

[slave1]
hostname=slave1.example.com
candidate_master=1

[slave2]
hostname=slave2.example.com
```

**Start Monitoring:**

```bash
# Check SSH connectivity
masterha_check_ssh --conf=/etc/mha/masterha_default.cnf

# Check replication health
masterha_check_repl --conf=/etc/mha/masterha_default.cnf

# Start MHA Manager
nohup masterha_manager --conf=/etc/mha/masterha_default.cnf &
```

## Backup Strategy

### Consistent Backups

**MySQL Backup with Percona XtraBackup:**

```bash
# Install XtraBackup
sudo apt install percona-xtrabackup-80

# Full backup (no locking, hot backup)
sudo xtrabackup --backup \
  --target-dir=/backups/mysql/full-$(date +%Y%m%d) \
  --user=root --password=password

# Prepare backup
sudo xtrabackup --prepare \
  --target-dir=/backups/mysql/full-$(date +%Y%m%d)

# Incremental backup
sudo xtrabackup --backup \
  --target-dir=/backups/mysql/inc-$(date +%Y%m%d-%H%M) \
  --incremental-basedir=/backups/mysql/full-$(date +%Y%m%d)

# Compress and upload to S3
sudo tar -czf - /backups/mysql/full-$(date +%Y%m%d) | \
  aws s3 cp - s3://backups/mysql/full-$(date +%Y%m%d).tar.gz
```

**PostgreSQL Backup:**

```bash
# Logical backup (pg_dump)
pg_dump -U postgres -F c -b -v -f \
  /backups/postgres/laravel-$(date +%Y%m%d).dump laravel

# Physical backup (pg_basebackup)
pg_basebackup -D /backups/postgres/base-$(date +%Y%m%d) \
  -F t -z -P -U replicator -h localhost

# Continuous WAL archiving
# postgresql.conf
archive_mode = on
archive_command = 'aws s3 cp %p s3://backups/postgres/wal/%f'
```

**Redis Backup:**

```bash
# Trigger RDB snapshot
redis-cli BGSAVE

# Wait for save to complete
redis-cli LASTSAVE

# Copy RDB file
cp /var/lib/redis/dump.rdb /backups/redis/dump-$(date +%Y%m%d).rdb

# Enable AOF for better durability
redis-cli CONFIG SET appendonly yes
```

### Retention Policy

```bash
# Daily backups for 7 days
# Weekly backups for 4 weeks
# Monthly backups for 12 months

# Cron job for retention
0 2 * * * find /backups -name "*.tar.gz" -mtime +7 -delete
0 3 * * 0 find /backups -name "weekly-*" -mtime +28 -delete
0 4 1 * * find /backups -name "monthly-*" -mtime +365 -delete
```

## Disaster Recovery

### Region Failover

**Multi-Region Setup:**

1. **Primary Region:** Active-active with synchronous replication
2. **Secondary Region:** Asynchronous replication
3. **Failover:** DNS update + connection pool flush

**DNS Failover:**

```bash
# Update Route53 health check
aws route53 update-health-check \
  --health-check-id <check-id> \
  --resource-path /health

# Failover to secondary
aws route53 change-resource-record-sets \
  --hosted-zone-id <zone-id> \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "db.example.com",
        "Type": "CNAME",
        "TTL": 60,
        "ResourceRecords": [{"Value": "db-secondary.example.com"}]
      }
    }]
  }'
```

**Recovery Steps:**

1. Verify secondary is up-to-date
2. Promote secondary to primary
3. Update application configuration
4. Flush connection pools
5. Update DNS
6. Monitor application health

### Point-in-Time Recovery

**MySQL:**

```bash
# Extract binary logs
mysqlbinlog --start-datetime="2025-02-07 10:00:00" \
  --stop-datetime="2025-02-07 11:00:00" \
  mysql-bin.000123 | mysql -u root -p
```

**PostgreSQL:**

```bash
# Restore from base backup
restore_command = 'aws s3 cp s3://backups/postgres/wal/%f %p'

# Recovery target
recovery_target_time = '2025-02-07 11:00:00'
recovery_target_inclusive = true
```

## Monitoring and Alerting

### Key Metrics

**MySQL:**
- Replication lag (Seconds_Behind_Master)
- Connection pool usage
- Query performance (slow query log)
- Binary log size

**PostgreSQL:**
- Replication lag (now() - pg_last_xact_replay_timestamp())
- WAL size and rate
- Connection pool usage
- Autovacuum activity

**Redis:**
- Memory usage
- Keyspace hits/misses
- Replication lag
- Cluster health

### Alert Thresholds

| Metric | Warning | Critical | Action |
|--------|---------|----------|--------|
| Replication lag | 5s | 30s | Check network, restart slave |
| Connection pool | 80% | 95% | Add more connections/servers |
| Disk usage | 80% | 90% | Clean up old backups |
| Master down | N/A | Immediate | Trigger failover |

## Maintenance Procedures

### Zero-Downtime Maintenance

**Rolling Updates:**

1. Update slave 1
2. Wait for replication to catch up
3. Promote slave 1 to master
4. Update old master (now slave)
5. Promote original master back

**Schema Migrations:**

```php
// Use online schema change tools
// pt-online-schema-change for MySQL
pt-online-schema-change \
  --alter "ADD COLUMN new_column INT" \
  --host=localhost --database=laravel --table=users \
  --execute

// pg_ghost for PostgreSQL
pg_ghost \
  --host=localhost --database=laravel --table=users \
  --alter "ADD COLUMN new_column INTEGER"
```

## Runbook

### Master Failure

1. Detect failure (monitoring alert)
2. Verify master is truly down
3. Promote healthiest slave to master
4. Update application configuration
5. Update DNS/load balancer
6. Monitor new master
7. Rebuild failed server as new slave

### Replication Lag

1. Check network connectivity
2. Check slave status (SHOW SLAVE STATUS)
3. Identify lagging transaction
4. Skip problematic transaction if necessary
5. Restart slave
6. Monitor for 30 minutes

### Split Brain

1. Stop all writes immediately
2. Identify most recent master
3. Demote others to slaves
4. Resynchronize all nodes
5. Resume writes
6. Investigate root cause

## Best Practices

1. **Test backups regularly** - Unrestored backups don't exist
2. **Monitor everything** - Set up comprehensive alerts
3. **Document procedures** - Create detailed runbooks
4. **Practice failover** - Conduct quarterly drills
5. **Use automation** - Eliminate manual steps
6. **Plan capacity** - Scale before hitting limits
7. **Secure replication** - Encrypt replication traffic
8. **Audit access** - Review user permissions regularly
